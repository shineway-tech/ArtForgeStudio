# 支付宝 iframe 扫码弹窗实现计划

> 依据已确认的设计文档：`docs/superpowers/specs/2026-07-16-alipay-iframe-checkout-design.md`。
> 当前为开发环境，直接在现有工作区实现；不提交代码，不调整支付宝回调、HTTPS 或后端支付产品。

## 目标

让积分充值和会员购买的支付宝付款页在桌面客户端中以与 `market_tool` 一致的嵌入式二维码形态展示，而不是把完整支付宝网页缩进二维码区域；同时精简弹窗并隐藏短暂的支付查询错误。

## 任务 1：补充失败回归测试

1. 扩展支付 UI 源码回归测试，要求 Wry 使用本地 HTML 而非直接加载支付 URL。
2. 要求弹窗不再包含订单摘要，并要求短暂查询失败不写入用户可见文案。
3. 运行聚焦测试，确认修改前测试失败。

## 任务 2：实现受控 iframe 包装页

1. 在 `native-client/src/runtime/payment_window.rs` 增加 HTML 属性转义和 iframe HTML 生成函数。
2. 支付地址仍先经过 HTTPS 和支付宝域名校验。
3. WebView 改用 `with_html` 加载本地包装页，包装页中的 220×220 iframe 指向转义后的支付宝地址。
4. 保留顶层导航拦截和新窗口拦截。
5. 增加 HTML 生成函数的单元测试。

## 任务 3：对齐紧凑弹窗布局

1. 移除订单摘要文本及相关状态字段。
2. 将弹窗高度缩短为 350px，只保留标题、二维码区域和状态。
3. 根据新布局修正 Wry 子 WebView 的垂直偏移，并更新 1x/2x 坐标测试。

## 任务 4：优化支付查询提示

1. 查询接口的短暂错误继续轮询，不覆盖“正在等待支付宝付款结果…”文案。
2. 达到重试上限或最终查询失败时，只展示友好提示，不暴露 `payment_query_failed` 等内部错误。
3. 保留支付成功、订单关闭和业务失败的明确终态提示。

## 任务 5：验证并重启客户端

1. 运行支付相关聚焦测试。
2. 运行 native-client 全量测试和 `cargo check`。
3. 重启本地客户端并确认进程保持运行。
4. 不提交代码。
