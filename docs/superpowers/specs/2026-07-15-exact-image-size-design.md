# 生图精确比例与清晰度尺寸传递设计

## 背景

当前客户端把 `1:1` 以外的多个比例压缩成 `landscape` 或 `portrait`，后端再固定向中转站传递 `1024x1024`、`1536x1024` 或 `1024x1536`。`2K` 和 `4K` 只影响上游 `quality`，图片返回后才由后端放大到目标最长边。

中转站根据生图请求中的 `size` 判断清晰度并计费，因此请求参数必须同时反映用户选择的真实比例与 `1K`、`2K`、`4K` 目标清晰度，不能继续使用固定三档尺寸。

## 目标

- 客户端向后端传递用户实际选择的比例，不再转换成横图、竖图或方图三类。
- 后端以服务端配置中的清晰度最长边为准，计算最终宽高并向中转站传递精确 `size`。
- `1K`、`2K`、`4K` 分别以最长边 `1024`、`2048`、`4096` 像素定义。
- 任务快照保存比例、目标宽高和实际上游 `size`，便于恢复任务、排查问题和核对中转站账单。
- 兼容已创建任务及旧客户端使用的 `square`、`landscape`、`portrait`。

## 非目标

- 不修改当前积分价格：`1K`、`2K`、`4K` 仍分别按现有档位扣费。
- 不修改图片模型名称、模型 ID、上游 `quality: low/medium/high` 映射。
- 不新增数据库列；尺寸数据写入现有 JSON 快照。
- 不调整前端比例选择器的视觉样式。

## 支持的比例

服务端接受以下精确比例：

`1:1`、`3:2`、`2:3`、`4:3`、`3:4`、`5:4`、`4:5`、`16:9`、`9:16`、`2:1`、`1:2`、`21:9`、`9:21`。

为兼容旧客户端和旧恢复记录，服务端继续接受以下别名：

| 旧值 | 兼容比例 |
| --- | --- |
| `square` | `1:1` |
| `landscape` | `3:2` |
| `portrait` | `2:3` |

新客户端只发送精确比例，不再发送旧别名。

## 尺寸算法

后端是目标尺寸的唯一权威计算方。客户端可以使用相同算法展示尺寸，但不能直接决定中转站计费尺寸。

1. 根据清晰度取得最长边：`1K = 1024`、`2K = 2048`、`4K = 4096`。
2. 解析比例为正整数 `ratio_width:ratio_height`。
3. 横图和方图令宽度等于最长边，按比例计算高度；竖图令高度等于最长边，按比例计算宽度。
4. 计算出来的短边四舍五入到最接近的 8 像素倍数，最小值为 64。
5. 最终中转站参数为字符串 `${width}x${height}`。

示例：

| 用户选择 | 目标尺寸 | 中转站 `size` |
| --- | ---: | --- |
| `1:1` + `1K` | 1024 × 1024 | `1024x1024` |
| `16:9` + `4K` | 4096 × 2304 | `4096x2304` |
| `9:16` + `4K` | 2304 × 4096 | `2304x4096` |
| `4:3` + `2K` | 2048 × 1536 | `2048x1536` |
| `21:9` + `4K` | 4096 × 1752 | `4096x1752` |

## 接口与数据流

### 客户端到后端

新客户端创建任务时直接发送选择值：

```json
{
  "task_type": "image_generation",
  "model_code": "openai_image",
  "quality": "4K",
  "aspect_ratio": "16:9"
}
```

客户端的普通生成与未完成任务恢复必须使用同一规则，避免首次提交和恢复提交产生不同的幂等请求内容。

### 后端任务创建

后端校验 `aspect_ratio` 是否属于精确比例或兼容别名，然后根据任务创建时的 `quality` 和模型价格配置计算尺寸。创建后的 `request_snapshot` 至少保存：

```json
{
  "aspect_ratio": "16:9",
  "target_width": 4096,
  "target_height": 2304,
  "provider_size": "4096x2304",
  "reference_file_ids": []
}
```

尺寸在任务创建时固定。Worker 直接读取快照，不因后续配置同步而改变已排队任务的中转站参数。

### 后端到中转站

无参考图时调用 `/images/generations`，有参考图时调用 `/images/edits`。两种请求都传递同一组尺寸参数：

```json
{
  "model": "gpt-image-2",
  "size": "4096x2304",
  "quality": "high",
  "n": 1
}
```

`size` 来自任务快照；`quality` 继续使用配置中的 `low`、`medium`、`high` 映射。

### 结果处理

中转站返回图片后，后端读取任务快照中的目标宽高。返回尺寸一致时不缩放；不一致时继续使用现有图片处理流程修正到目标宽高，再上传 OSS。这样最终文件元数据与请求快照保持一致，但中转站计费依据始终是已经精确传递的 `size`。

## 配置与模型目录

- 开发配置和开发示例配置中的图像模型 `capabilities.aspect_ratios` 改为完整精确比例列表；本次不修改生产环境配置。
- 因模型配置哈希发生变化，开发环境同步时同时提升 `models.catalog_version` 与图像模型内部 `version`，避免触发 `model_catalog_version_conflict`。展示名称和价格保持不变。
- `max_long_edge` 继续作为 `1K`、`2K`、`4K` 的最长边来源，不在代码中重复维护另一份计费尺寸配置。
- 模型目录接口向客户端返回精确比例能力，后续客户端可以据此限制可选比例。

## 向后兼容

- 新接口接受三个旧别名，确保旧客户端仍能创建任务。
- 对已经创建且快照中没有 `provider_size` 的任务，Worker 根据旧别名和任务的 `pricing_snapshot.max_long_edge` 计算兼容尺寸。
- 旧别名只用于兼容，不重新写回历史任务。
- 精确比例参与幂等请求身份计算。同一个 `client_request_id` 使用不同精确比例时仍返回冲突，避免错误复用任务。

## 错误处理

- 不支持的比例在任务创建阶段返回 `400 validation_failed`，不得进入队列或预占积分。
- 清晰度没有有效 `max_long_edge` 时返回配置错误，不向中转站发送请求。
- 计算尺寸必须为正整数且不超过服务端配置允许范围。
- 中转站拒绝精确 `size` 时保留其失败码进入现有任务失败流程，未生成图片的积分按现有结算逻辑释放。

## 测试设计

### 客户端

- 普通生成请求保留 `16:9`、`4:3`、`21:9` 等精确比例，不再发送 `landscape`。
- 竖图比例同样保留原值。
- 恢复任务提交与普通提交使用相同精确比例。
- 现有尺寸预览测试继续验证最长边和按 8 像素取整规则。

### 后端

- 尺寸计算单元测试覆盖 13 种比例和三个清晰度档位。
- 单元测试覆盖三个旧别名的兼容结果。
- 路由校验接受完整比例列表并拒绝未知值、零值和任意字符串。
- 上游服务测试断言 `/images/generations` 与 `/images/edits` 收到精确 `size`。
- 任务创建测试断言 `request_snapshot` 保存目标宽高和 `provider_size`。
- Worker 测试断言优先读取任务快照，并能兼容没有新字段的旧任务。
- 跨端测试覆盖客户端请求中的比例与后端任务快照、上游请求尺寸一致。

## 验收标准

- 选择任意受支持比例和清晰度后，中转站收到的 `size` 等于按本文算法计算的目标尺寸。
- `16:9` 不再被发送为 `1536x1024`，`9:16` 不再被发送为 `1024x1536`，除非该尺寸恰好是算法结果。
- 任务详情、结果文件宽高和任务快照能够对应同一个目标尺寸。
- 旧客户端和已有未完成任务可以继续执行。
- 所有新增单元测试、跨端测试、后端测试、客户端测试及编译检查通过。
