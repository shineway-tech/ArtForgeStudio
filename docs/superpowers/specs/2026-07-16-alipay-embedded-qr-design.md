# 支付宝网站支付应用内二维码设计

## 目标

会员购买、会员升级和积分充值继续使用支付宝电脑网站支付能力，但不再打开系统浏览器或完整支付宝收银台。应用调用 `alipay.trade.page.pay` 时启用支付宝的嵌入式二维码模式，并在 ArtForgeStudio 主窗口内展示支付宝托管的扫码页面。

支付成功必须继续由服务端异步通知或主动查单确认。客户端页面加载、二维码显示和前端跳转均不能作为到账依据。

## 不在本次范围内

- 不切换到 `alipay.trade.precreate`，因此不要求新增“当面付”权限。
- 不把支付宝签名 URL 自行编码成二维码。
- 不抓取、解析或重写支付宝返回的 HTML。
- 不修改订单、支付流水或积分账本的数据库结构。
- 不进行真实付款测试，不创建 Git 提交。

## 参考实现

参考 `/Users/fanxiao/workstation/market_tool/backend/channel_nest_api/src/services/alipay.js`：

- 使用 `alipay.trade.page.pay`；
- `product_code` 为 `FAST_INSTANT_TRADE_PAY`；
- `qr_pay_mode` 为 `4`；
- `qrcode_width` 为 `220`；
- 将支付宝签名页面加载到应用内嵌页面；
- 通过服务端查单和异步通知确认支付。

ArtForgeStudio 保留现有订单恢复、支付通知验签、主动查单、超时关单和权益发放逻辑，只替换支付页面的展示方式。

## 后端设计

### 支付请求

`src/services/alipay.js` 的网站支付请求增加：

```json
{
  "product_code": "FAST_INSTANT_TRADE_PAY",
  "qr_pay_mode": "4",
  "qrcode_width": 220
}
```

保留 `notify_url` 和订单超时时间。应用不依赖同步跳转，因此嵌入式二维码模式不发送 `return_url`，避免支付完成后的同步页面跳出允许的导航范围。

会员购买、会员续费、会员升级和积分充值共用 `initializePayment`，因此全部使用相同二维码模式。

### API 和持久化

现有订单响应保持兼容：

```json
{
  "payment": {
    "provider": "alipay",
    "channel": "website",
    "status": "waiting_buyer",
    "checkout_url": "https://openapi.alipay.com/gateway.do?...",
    "checkout_expires_at": "..."
  }
}
```

`checkout_url` 仍保存在 `payment_transactions.provider_payload`，仅在订单待支付且未过期时返回。数据库不新增字段、不执行迁移。

不得在普通日志、错误消息或通知中输出完整 `checkout_url`，因为其中包含支付宝签名参数。

## 客户端设计

### 应用内展示

`payment_window.rs` 在 macOS 和 Windows 上统一使用 Wry 子 WebView，并挂载到 Slint 主窗口。macOS 不再调用系统浏览器。

WebView 显示在主窗口中央的支付区域，支付宝页面通过 `qr_pay_mode=4` 只呈现可自定义宽度的嵌入式二维码。客户端不生成二维码图片，也不修改支付宝页面内容。

客户端初始 URL 只接受 HTTPS，并限制为以下支付宝支付域名：

- `openapi.alipay.com`
- `openapi-sandbox.dl.alipaydev.com`

后续导航只允许初始支付域名及受信任的 `alipay.com`、`alipayobjects.com`、`alipaydev.com` 子域；拒绝 HTTP、外部域名、下载和新窗口跳转。

### 支付状态

创建订单并成功打开二维码后：

- 客户端显示“请使用支付宝扫一扫完成支付”；
- 每 3 秒调用现有订单同步接口；
- 支付已确认但权益仍在发放时显示“付款已确认，正在等待权益到账”；
- 权益发放完成后关闭 WebView，关闭购买弹窗，刷新会员、积分、通知和订单恢复记录；
- 订单关闭或过期后关闭 WebView，并允许重新发起订单。

用户手动关闭二维码只隐藏支付页面，不把订单标记为成功，也不删除服务端订单。后台轮询和启动恢复仍可确认后续到账；恢复到有效待支付订单时可重新展示原 `checkout_url`。

### 文案

积分和会员入口统一使用以下语义：

- 标题：`支付宝扫码支付`
- 说明：`请使用支付宝扫一扫完成支付，支付结果以服务端确认为准。`
- 创建按钮：`生成支付二维码`
- 等待状态：`正在等待支付宝付款结果...`
- 加载失败：`支付二维码加载失败，请关闭后重试。`

不再出现“打开支付宝”“打开收银台”或“系统浏览器启动失败”。

## 数据流

1. 客户端提交会员或积分订单。
2. 后端创建业务订单和支付流水。
3. 后端使用 `alipay.trade.page.pay` 生成带 `qr_pay_mode=4` 的签名 URL。
4. 客户端验证域名并在主窗口子 WebView 中加载 URL。
5. 用户使用手机支付宝扫码支付。
6. 支付宝向后端发送异步通知；客户端同时定期请求主动查单。
7. 后端验签、核对应用、商户、金额和订单号后标记已支付并发放会员或积分。
8. 客户端取得 `paid + fulfilled` 状态后关闭二维码并刷新账户。

## 错误与恢复

- 支付宝未返回有效 HTTPS URL：后端返回 `payment_initialization_failed`，支付流水记录安全的错误码。
- 客户端收到非支付宝域名：拒绝加载并显示二维码加载失败，不回退到外部浏览器。
- WebView 初始化失败：保留待支付订单和恢复记录，停止当前展示，不把页面失败误判为支付失败。
- 查单暂时失败：保留二维码，继续按现有上限重试。
- 支付成功但权益发放重试：显示等待到账，不能把订单降级为失败或待支付。
- 订单过期：后端查单后关单，客户端关闭二维码并提示重新发起。
- 客户端退出：本地待处理订单记录保留；下次启动查询原订单并恢复有效支付页面或到账结果。

- 运行后端全量测试、lint 和配置检查。

### 客户端

- 先添加失败测试，断言初始支付地址必须是支付宝 HTTPS 域名。
- 测试允许支付宝受信任跳转，拒绝 HTTP、伪造后缀和外部域名。
- 测试支付弹窗不再包含外部收银台文案。
- 测试订单完成、权益发放中、关闭和过期状态。
- 在 macOS 运行单元测试和 `cargo check`，确认 Wry/WKWebView 能编译。
- 保留 Windows WebView2 代码路径，确保条件编译不会退回外部浏览器。

### 集成验证

- 使用可注入的测试构造器验证应用内 WebView 打开参数和边界计算，不加载真实或伪造的支付域名。
- 使用 Mock API 验证积分订单和会员订单都返回 `checkout_url` 并进入轮询。
- 手动确认客户端不会打开系统浏览器。

## 验收标准

- 会员和积分支付都使用 `alipay.trade.page.pay` 的嵌入式二维码模式。
- macOS 和 Windows 客户端均在应用内展示支付宝二维码，不打开外部浏览器。
- 支付结果只由后端通知或主动查单确认。
- 支付成功后会员或积分正确到账，二维码自动关闭。
- 无新增数据库迁移，无真实支付测试，无代码提交。
