# 积分明细分页与可读性改造设计

## 目标

积分明细必须支持逐页浏览，并将后端流水字段转换成普通用户能理解的中文业务信息。页面不得直接展示 `generation_task`、RFC 3339 时间或“结算 0 积分”等实现细节。

## 分页方案

- 复用后端 `/v1/credits/ledger` 已有的 `cursor`、`limit` 和 `meta.next_cursor`。
- 每页显示 8 条记录，页面底部显示“上一页 / 第 N 页 / 下一页”。
- 第一页禁用“上一页”，没有 `next_cursor` 时禁用“下一页”。
- 客户端保存每一页的起始游标，使用户可以前后翻页；不预先加载全部流水。
- 翻页期间禁用按钮，避免重复请求。请求失败时保留当前页并展示错误提示。

## 流水展示规则

| 后端类型 | 用户标题 | 金额展示 | 说明 |
| --- | --- | --- | --- |
| `reserve` | AI 创作积分暂时冻结 | 冻结 N | 任务处理中暂时冻结，失败或未使用部分会自动退回 |
| `commit` | AI 创作积分已扣除 | 扣除 N | 任务完成，已从冻结积分中结算 |
| `release` | 未使用积分已退回 | 退回 N | 未消耗的冻结积分已返还 |
| `grant` | 优先使用后端业务描述 | +N | 展示对应的中文业务来源 |
| `expire` | 积分已过期 | -N | 展示对应的中文业务来源 |

`commit` 的可用积分变化通常为 0，因此金额使用 `reserved_delta` 的绝对值，避免向用户显示“积分结算 0”。冻结使用中性色，扣除或过期使用红色，退回或赠送使用绿色。

业务类型转换为中文：`generation_task` 为“AI 创作”、`generation_retry` 为“任务重试”、`membership` 为“会员赠送”、`membership_upgrade` 为“会员升级”、`order` 为“积分充值”、`registration` 为“注册赠送”、`user` 为“人工调整”、`outbox_event` 为“系统补偿”，未知类型显示“积分变动”。

时间转换为本地 `YYYY-MM-DD HH:mm` 格式。每条记录补充变动后的可用积分余额，帮助用户理解流水结果。

## 客户端结构

- API 层新增分页结果对象，保留流水列表和下一页游标。
- 运行时 Store 保存当前页、页游标历史和下一页游标。
- 独立的积分回调模块负责翻页请求、状态切换和错误处理。
- 独立的格式化函数负责把后端流水转换成 UI 展示模型，Slint 页面只负责渲染。

## 验证

- 单元测试覆盖冻结、结算、退回和未知业务类型的文案与金额转换。
- 单元测试确保页面中不再出现 `generation_task`，并验证时间格式。
- 编译检查验证 Rust 与 Slint 的属性、回调和模型字段一致。
- 启动本地客户端，验证第一页按钮状态、前后翻页和现有登录状态不受影响。
