# 失败生成记录删除设计

## 目标

在生成结果列表中，为失败记录提供删除入口。用户把鼠标悬停在失败卡片上时，卡片右上角显示垃圾桶按钮；点击后复用现有删除确认弹窗。

## 范围

- 仅处理客户端本地 `source_path == "failed"` 的生成失败记录。
- 确认删除后，从本地 `generations` 集合移除对应记录，保存本地数据并刷新生成结果列表。
- 不删除服务端任务历史，不改变积分、通知、成功作品或正在执行的生成任务。
- 成功图片现有的删除、拖拽和查看器交互保持不变。

## 交互设计

失败卡片默认保持当前外观和“重试”按钮。鼠标进入卡片时，右上角显示与成功图片一致的垃圾桶图标；离开后隐藏。

点击垃圾桶只触发删除确认，不触发“重试”，也不打开查看器。确认弹窗继续使用现有中英文文案和取消/删除操作。用户确认后删除失败记录；取消或关闭弹窗时不修改数据。

## 实现设计

改动集中在 `ThumbnailCard`：

1. 为失败卡片增加独立的悬停触摸区域。
2. 复用现有垃圾桶图标、尺寸、位置和危险态配色。
3. 复用 `AppState.request-delete-thumbnail(item.id, "generation")`。
4. 继续由现有 `confirm-delete` 回调负责更新本地 Store、持久化并刷新界面。

不增加服务端接口，不改变本地数据结构，也不新增删除状态。

## 边界与错误处理

- 删除入口只在失败卡片上悬停时出现。
- 失败卡片继续禁止拖拽和打开查看器。
- “重试”和“删除”使用独立点击区域，互不触发。
- 如果记录在确认前已经不存在，现有 `retain` 删除逻辑保持幂等，刷新后界面自然收敛。

## 测试

新增或扩展客户端源码契约测试，验证：

- 失败卡片拥有悬停区域。
- 悬停时显示右上角垃圾桶按钮。
- 点击删除入口调用 `request-delete-thumbnail`，来源为 `generation`。
- 失败卡片仍不参与拖拽或查看器打开流程。
- 现有删除确认回调和成功图片删除行为不回归。

最后运行客户端完整测试与编译检查。
